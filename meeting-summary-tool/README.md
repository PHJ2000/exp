# Meeting Summary Tool

로컬 우선 회의 요약 도구 기획 프로젝트입니다. 회의 녹음 파일을 넣으면 로컬에서 텍스트로 바꾸고, 가능하면 화자까지 나눈 뒤, 회의 요약과 액션 아이템을 Markdown으로 저장하는 도구를 목표로 합니다.

작성일: 2026-04-11

## 목적

비싼 회의 요약 서비스를 매번 구독하지 않고도, 팀 내부에서 먼저 실사용해볼 수 있는 회의 녹음, 전사, 화자 분리, 요약 도구를 만드는 것이 목표입니다.

핵심 방향은 다음과 같습니다.

- 실시간 처리는 MVP의 필수 조건이 아닙니다.
- 초기에는 팀 내부에서 직접 쓸 수 있으면 됩니다.
- 가능한 무료에 가깝게 동작해야 합니다.
- 다양한 회의 환경에서 쉽게 써야 합니다.
- 화자 분리는 최종적으로 안정화하고 싶지만, MVP의 첫 번째 성공 조건은 아닙니다.
- 처음부터 서버비가 큰 hosted 서비스로 시작하지 않습니다.

## 한 줄 설명

회의 녹음 파일을 넣으면 로컬에서 전사하고, 가능하면 화자를 나눈 뒤, 회의 요약과 결정사항, 액션 아이템을 Markdown으로 저장하는 도구.

## 현재 결론

초기 제품은 "한 사람이 녹음 파일을 넣는 방식"으로 시작합니다.

가장 현실적인 MVP 흐름은 다음과 같습니다.

```text
회의 녹음 파일
→ 로컬 STT
→ 선택적 화자 분리
→ Speaker 1/2/3 transcript 생성
→ 사용자가 Speaker 이름 보정
→ 요약/결정사항/액션 아이템 생성
→ Markdown 저장
```

실시간 화자 분리나 참가자별 접속은 초기 MVP에 넣지 않고, 나중에 정확도 모드로 추가합니다.

## 왜 이 방향인가

참가자별 접속 방식은 정확도가 좋지만, 모든 참석자가 링크에 들어와 권한을 허용해야 하므로 초기 사용성이 무겁습니다. 반대로 한 사람이 녹음 파일을 넣는 방식은 정확도 한계가 있어도 사용성이 좋아 팀 내부에서 가치를 빨리 검증하기 좋습니다.

따라서 로드맵은 아래처럼 가져갑니다.

```text
기본 모드:
한 사람이 녹음/업로드
→ 로컬 후처리
→ Speaker 이름 보정
→ 요약 저장

정확도 모드:
참가자들이 링크로 접속
→ 참가자별 오디오 트랙 저장
→ 사용자 ID 기반 전사
→ 화자 분리는 보조 기능으로 사용
```

## MVP 범위

반드시 들어갈 기능:

- 오디오 파일 업로드
- `mp3`, `wav`, `m4a` 같은 일반 오디오 포맷 지원
- 로컬 STT로 전체 transcript 생성
- 요약 생성
- 결정사항 추출
- 액션 아이템 추출
- Markdown 파일 export
- 처리 결과 로컬 저장

가능하면 넣을 기능:

- Speaker 1/2/3 형태의 화자 분리
- Speaker 이름 수동 보정
- 회의 제목, 날짜, 참석자 메모
- transcript와 summary를 같은 파일에 저장

이번 단계에서 제외할 기능:

- 실시간 전사
- 실시간 화자 분리
- 참가자별 웹 접속
- Zoom/Meet/Teams 직접 연동
- 자동 캘린더 연동
- 서버 기반 대규모 처리
- 완전 자동 실명 화자 식별

## 초기 사용자 흐름

```text
1. 회의를 녹음한다.
2. 회의가 끝난 뒤 파일을 앱에 넣는다.
3. 앱이 로컬에서 전사를 수행한다.
4. 가능하면 화자 분리도 수행한다.
5. 사용자가 Speaker 1/2/3을 실제 이름으로 바꾼다.
6. 앱이 회의 요약, 결정사항, 액션 아이템을 만든다.
7. Markdown으로 저장한다.
```

## 출력 예시

```markdown
# 2026-04-11 팀 회의

## 요약

- 이번 회의에서는 MVP 범위를 파일 업로드, STT, 요약, Markdown 저장으로 잡았다.
- 화자 분리는 v1 기능으로 두고, 초기에는 수동 보정을 허용한다.

## 결정사항

- 실시간 처리는 MVP에서 제외한다.
- 로컬 처리 우선으로 구현한다.

## 액션 아이템

- pjh: 샘플 녹음 파일 3개를 준비한다.
- 팀원 A: 출력 Markdown 포맷에 대한 피드백을 준다.

## Transcript

Speaker 1: 오늘은 회의록 도구 MVP 범위를 정해봅시다.
Speaker 2: 실시간은 없어도 되고, 녹음 파일 업로드부터 하면 될 것 같습니다.
```

## 권장 기술 방향

현재 기준 기본 방향:

- v0, v1의 코어 구현 언어는 Python으로 둡니다.
- v0 실행 형태는 CLI로 시작합니다.
- 로컬 웹앱은 v1 또는 후속 단계 확장 후보로 둡니다.
- Rust는 초기 코어 대체 언어가 아니라, 나중에 UI, 런처, 패키징 계층에서 검토합니다.

### STT

우선 후보:
- `faster-whisper`
- `whisper.cpp`

초기 권장:
- Python 프로토타입: `faster-whisper`
- 나중에 로컬 앱 배포: `whisper.cpp` 검토

### 화자 분리

우선 후보:
- `pyannote.audio`
- 필요 시 NVIDIA NeMo
- 비교용 관리형 API: OpenAI, AssemblyAI, Deepgram

초기 권장:
- v0: STT + 요약 먼저
- v1: `pyannote.audio` 기반 화자 분리 추가

### 요약

우선 후보:
- 로컬 LLM: Ollama 기반 모델
- BYOK 방식: 사용자가 직접 OpenAI API 키 입력
- 초기 개발 편의용: 개발자 개인 API 키 테스트

초기 권장:
- 팀 내부: 개발자 API 키 또는 개인 Ollama
- 무료 배포: Ollama 우선 + BYOK 옵션

### 저장

초기 권장:
- v0: Markdown export
- v1: SQLite + Markdown export

### UI

초기 권장:
- v0: CLI 또는 간단한 로컬 웹앱
- v1: 로컬 웹앱
- v2: Tauri 또는 Electron 검토
