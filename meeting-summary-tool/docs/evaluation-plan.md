# 평가 계획

이 문서는 `meeting-summary-tool`의 초기 MVP를 검증하기 위한 최소 평가 기준을 정리한 초안이다.  
목적은 기능을 많이 늘리는 것이 아니라, 실제 회의 파일 하나를 넣었을 때 끝까지 처리되는지와 결과물이 실사용 가능한 수준인지 확인하는 데 있다.

## 1. 평가 대상

초기 평가는 다음 흐름이 한 번에 동작하는지를 본다.

```text
오디오 파일 입력
→ 전사
→ 필요 시 화자 분리
→ 요약 / 결정사항 / 액션 아이템 생성
→ Markdown 저장
```

평가 대상은 UI 완성도가 아니라 결과 품질과 처리 안정성이다.

## 2. 샘플 오디오 확보 기준

샘플은 최소 3개 이상 확보한다.  
가능하면 회의 환경 차이를 반영할 수 있도록 서로 다른 조건을 포함한다.

### 필수 기준

- 실제 회의 녹음에 가까운 파일이어야 한다.
- 파일 길이는 최소 5분 이상을 권장한다.
- 포맷은 `mp3`, `wav`, `m4a` 중 하나 이상을 포함한다.
- 사람이 실제로 듣고 발화 주제를 대략 확인할 수 있어야 한다.
- 파일명과 메타정보로 녹음 상황을 구분할 수 있어야 한다.

### 권장 조합

- 단일 화자에 가까운 파일 1개
- 2인 이상 대화가 섞인 파일 1개
- 잡음이 조금 있는 실사용 환경 파일 1개

### 제외 기준

- 너무 짧아서 회의 구조가 드러나지 않는 파일
- 음악, 효과음, 비음성 위주 파일
- 사생활 이슈로 공유가 불가능한 파일
- 내용 확인이 불가능한 저품질 녹음

### 샘플 메모 항목

각 샘플에는 아래 정도의 메모를 붙인다.

- 녹음 길이
- 참여 인원 추정
- 마이크 또는 녹음 환경
- 배경 잡음 수준
- 회의 성격: 정기회의, 스탠드업, 기획회의 등

## 3. MVP 성공 기준

MVP는 아래 조건을 만족하면 성공으로 본다.

### 기능 기준

- 파일 업로드 또는 입력이 가능하다.
- 전사가 끝까지 완료된다.
- 요약이 생성된다.
- 결정사항과 액션 아이템이 분리되어 나온다.
- 결과가 Markdown으로 저장된다.
- 저장된 파일을 그대로 열어 사람이 읽을 수 있다.

### 품질 기준

- 전사 결과가 원문 회의 흐름을 대략 따라간다.
- 요약이 회의 주제와 어긋나지 않는다.
- 결정사항과 액션 아이템이 빈 칸만 채워진 형태로 나오지 않는다.
- 사람이 이름 보정이나 문장 손질을 조금만 하면 바로 재사용할 수 있다.

### 운영 기준

- 한 번 실행해서 끝까지 결과를 얻을 수 있다.
- 실패가 나면 어디서 막혔는지 확인할 수 있다.
- 같은 입력을 다시 돌렸을 때 결과 구조가 크게 흔들리지 않는다.

## 4. 평가 관점

평가는 전사, 요약, 처리 시간 세 축으로 나눠 본다.

### 4.1 전사 평가

전사는 아래를 본다.

- 발화 구간이 대체로 맞는지
- 문장 단위가 너무 자주 끊기지 않는지
- 고유명사와 숫자가 어느 정도 유지되는지
- 화자 분리가 있으면 Speaker 구분이 크게 무너지지 않는지

전사 품질은 완전 정답 일치보다도, 사람이 후처리할 수 있는 수준인지가 중요하다.

### 4.2 요약 평가

요약은 아래를 본다.

- 회의 주제가 한눈에 드러나는지
- 논의한 핵심 쟁점이 빠지지 않았는지
- 결론과 보류 사항이 구분되는지
- 액션 아이템이 누가 무엇을 할지까지 이어지는지

요약은 길이보다 구조를 우선한다.  
짧아도 회의 의미가 유지되면 통과로 본다.

### 4.3 처리 시간 평가

처리 시간은 사용자가 기다릴 수 있는 수준인지 확인한다.

- 작은 파일은 빠르게 끝나야 한다.
- 긴 파일은 진행 상태를 확인할 수 있어야 한다.
- 전사, 요약, 저장 단계별로 어느 구간에서 시간이 쓰였는지 확인할 수 있어야 한다.

기준값은 나중에 실제 구현 성능을 보고 다시 정한다.  
현재 단계에서는 절대 시간보다 체감 대기와 단계 분리가 더 중요하다.

## 5. 최소 검증 절차

초기 검증은 아래 순서로 진행한다.

1. 샘플 오디오 3개를 준비한다.
2. 각 샘플의 길이와 환경 메모를 적는다.
3. 하나의 샘플을 입력해서 전사가 완료되는지 확인한다.
4. 같은 샘플로 요약과 액션 아이템이 생성되는지 확인한다.
5. Markdown 저장본을 열어 제목, 요약, 결정사항, 액션 아이템 구조가 유지되는지 확인한다.
6. 전사 결과와 저장 결과를 사람이 1회 검토한다.
7. 다른 샘플 2개에도 같은 절차를 반복한다.

### 통과 판단

아래 조건을 만족하면 초기 검증을 통과로 본다.

- 3개 샘플 모두 끝까지 처리된다.
- 결과 Markdown의 구조가 깨지지 않는다.
- 전사와 요약이 모두 사람이 손볼 수 있는 수준이다.
- 처리 중 어느 단계에서 문제가 났는지 확인 가능하다.

## 6. 후속 조정 포인트

초기 검증 뒤에는 아래 항목을 다시 본다.

- 화자 분리를 MVP에 계속 둘지
- 전사 모델을 유지할지 교체할지
- 요약 프롬프트를 고정할지 템플릿화할지
- 처리 시간 목표를 명시할지
- 샘플 오디오 기준을 더 좁힐지

이 문서는 구현이 진행되면 실제 결과에 맞춰 갱신한다.
