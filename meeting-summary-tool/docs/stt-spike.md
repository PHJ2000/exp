# STT 스파이크 초안

작성일: 2026-04-11

이 문서는 `meeting-summary-tool`의 Python 기준 로컬 STT 스파이크 범위를 정리한 초안입니다.  
목적은 `faster-whisper`를 중심으로, 회의 녹음 파일을 로컬에서 전사하는 최소 파이프라인을 빠르게 검증하는 것입니다.

## 1. 스파이크 목표

- 로컬 오디오 파일을 받아 전사 텍스트를 얻는다.
- 전사 결과를 세그먼트 단위로 확인한다.
- 요약 단계로 넘길 수 있는 중간 구조를 만든다.
- CPU 환경에서도 기본 동작 가능 여부를 확인한다.
- GPU가 있으면 처리 속도와 품질이 얼마나 달라지는지 비교한다.

## 2. 스파이크 범위

이번 스파이크에서 확인할 것:

- Python에서 `faster-whisper`를 설치하고 실행할 수 있는지
- 회의 녹음 파일을 입력받아 전사까지 이어지는지
- 길이가 긴 오디오에서 세그먼트가 안정적으로 잘리는지
- 전사 결과를 이후 요약 단계가 읽을 수 있는 구조로 정리할 수 있는지
- CPU와 GPU에서 모델 크기별 체감 차이가 있는지

이번 스파이크에서 보류할 것:

- 화자 분리
- 실시간 전사
- 스트리밍 입력
- 캘린더 연동
- 화자 이름 보정 UI
- 최종 Markdown 저장 포맷 확정

## 3. 지원 입력 포맷

우선 지원 후보는 아래와 같습니다.

- `wav`
- `mp3`
- `m4a`
- `mp4`
- `flac`
- `aac`

우선순위는 다음과 같이 둡니다.

1. 회의 녹음에서 흔한 `wav`, `mp3`, `m4a`
2. 영상 파일에 섞여 들어오는 `mp4`
3. 품질 확인용 `flac`, `aac`

스파이크 단계에서는 입력 포맷을 직접 엄격하게 제한하기보다, `faster-whisper`가 읽을 수 있는지 확인하는 쪽에 집중합니다.  
다만 최종 제품에서는 파일 확장자와 실제 디코딩 가능 여부를 분리해서 검증하는 편이 안전합니다.

## 4. 호출 흐름

기본 호출 흐름은 다음처럼 잡습니다.

```text
오디오 파일 경로 입력
→ 파일 존재 및 확장자 확인
→ 전사 옵션 구성
→ faster-whisper 모델 로드
→ 오디오 전사 실행
→ segment / text / timestamps 수집
→ 중간 구조로 정리
→ 다음 단계(요약, 저장)로 전달
```

실행 단위는 아래 둘 중 하나로 시작하는 것이 현실적입니다.

- CLI 한 번 실행으로 `input -> transcript.json -> transcript.md`
- 내부 함수 호출로 `transcribe_audio(path, options) -> TranscriptBundle`

스파이크에서는 CLI가 더 단순합니다. 이후 로컬 웹앱으로 넘어가더라도 전사 코어는 함수 단위로 분리해두는 편이 좋습니다.

## 5. 전사 결과 중간 구조 초안

요약 단계로 넘기기 전에, 전사 결과는 문자열 하나만 두지 않고 구조화해두는 편이 좋습니다.  
초안 구조는 아래와 같습니다.

```json
{
  "source_file": "path/to/meeting.m4a",
  "language": "ko",
  "duration_sec": 3721,
  "model": "small",
  "device": "cpu",
  "segments": [
    {
      "id": 0,
      "start": 0.0,
      "end": 4.2,
      "text": "오늘 회의는 MVP 범위를 정하는 게 목적입니다.",
      "avg_logprob": -0.18,
      "no_speech_prob": 0.02
    },
    {
      "id": 1,
      "start": 4.2,
      "end": 9.8,
      "text": "우선 로컬 STT와 Markdown 저장부터 확인하죠.",
      "avg_logprob": -0.21,
      "no_speech_prob": 0.01
    }
  ],
  "full_text": "오늘 회의는 MVP 범위를 정하는 게 목적입니다. ...",
  "warnings": []
}
```

최소 필수 필드는 아래 정도가 적당합니다.

- `source_file`
- `language`
- `duration_sec`
- `model`
- `device`
- `segments`
- `full_text`
- `warnings`

`segments`는 요약 프롬프트, 재생 위치 이동, 품질 확인에 모두 활용할 수 있어야 합니다.  
나중에 화자 분리가 붙더라도 이 구조 위에 `speaker` 필드만 얹을 수 있도록 설계하는 편이 좋습니다.

## 6. 모델 및 옵션 후보

`faster-whisper` 기준 후보는 아래처럼 검토합니다.

### 모델 크기 후보

- `tiny`
- `base`
- `small`
- `medium`
- `large-v3`

### 검토 기준

- `tiny`, `base`: CPU 초동 검증용
- `small`: 기본 MVP 후보
- `medium`: 정확도 확인용
- `large-v3`: 품질 상한 확인용, GPU 여부와 함께 검토

### 주요 옵션 후보

- `device`: `cpu`, `cuda`
- `compute_type`: `int8`, `int8_float16`, `float16`, `float32`
- `language`: `ko` 고정 여부 또는 자동 감지
- `beam_size`
- `best_of`
- `vad_filter`
- `condition_on_previous_text`
- `temperature`

스파이크 단계에서는 다음 조합을 먼저 보는 것이 무난합니다.

- CPU 기본: `small` + `int8`
- GPU 기본: `small` 또는 `medium` + `float16`
- 정확도 비교: `base`, `small`, `medium` 샘플 비교

## 7. CPU / GPU 고려사항

### CPU 고려사항

- 설치가 쉬워야 합니다.
- 메모리가 작은 환경에서도 최소 동작해야 합니다.
- 긴 파일은 처리 시간이 길어질 수 있으므로 진행 상태 표시가 필요합니다.
- `int8` 계열 양자화가 유리할 가능성이 큽니다.

CPU 환경에서는 모델 품질보다 실행 가능성이 먼저입니다.  
따라서 스파이크의 1차 성공 기준은 "오디오를 끝까지 처리할 수 있는가"입니다.

### GPU 고려사항

- NVIDIA CUDA 사용 가능 여부를 먼저 봅니다.
- `float16` 또는 `int8_float16` 조합이 유력합니다.
- 대형 모델을 쓰면 정확도는 좋아질 수 있지만 메모리 요구량이 커집니다.
- 전사 시간과 품질 개선 폭을 함께 봐야 합니다.

GPU 환경에서는 "더 빠른가"만 보지 말고, 같은 샘플에서 오탈자, 끊김, 누락이 얼마나 줄어드는지도 같이 확인합니다.

## 8. 실패 처리 포인트

스파이크 단계에서 반드시 확인할 실패 지점은 다음과 같습니다.

- 입력 파일 없음
- 지원하지 않는 파일 형식
- 파일은 존재하지만 디코딩 실패
- 모델 다운로드 실패
- 로컬 캐시 경로 문제
- 메모리 부족
- GPU 미지원 또는 CUDA 초기화 실패
- 너무 긴 파일로 인한 과도한 처리 시간
- 언어 자동 감지가 틀리는 경우
- 전사 도중 세그먼트가 비정상적으로 비는 경우

권장 처리 방식:

- 사용자에게 실패 원인을 짧고 분명하게 보여준다.
- 전사 전체 실패와 세그먼트 일부 실패를 구분한다.
- 가능한 경우 경고를 남기고 partial result를 보존한다.
- 모델 다운로드 실패는 다시 시도할 수 있게 한다.
- 디코딩 실패는 파일 자체 문제로 분류한다.

스파이크에서 특히 중요한 건 "실패했을 때 아무 것도 남지 않는 상태"를 피하는 것입니다.  
전사가 일부만 성공해도, 어디까지 성공했는지 남겨야 다음 단계에서 디버깅이 쉽습니다.

## 9. 스파이크 산출물

이번 스파이크의 예상 산출물은 아래입니다.

- 실행 가능한 Python 전사 함수 초안
- 입력 파일 경로를 받아 전사 결과를 반환하는 인터페이스
- `TranscriptBundle` 같은 중간 구조 초안
- CPU 기준 샘플 결과
- GPU 기준 샘플 결과 또는 검증 메모
- 실패 케이스 메모
