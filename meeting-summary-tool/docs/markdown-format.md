# 회의 요약 결과 Markdown 형식

이 문서는 `meeting-summary-tool`이 생성하는 회의 요약 결과의 Markdown 출력 규격을 정리합니다.

목표는 다음 세 가지입니다.

- 사람이 바로 읽을 수 있어야 합니다.
- 이후 자동화 처리나 검색 인덱싱에 써도 구조가 흔들리지 않아야 합니다.
- 전사 결과와 요약 결과를 한 파일 안에서 함께 다룰 수 있어야 합니다.

## 기본 원칙

- 출력은 기본적으로 단일 Markdown 파일 1개를 기준으로 합니다.
- 파일 상단에는 메타데이터를 둡니다.
- 본문은 `요약`, `결정사항`, `액션 아이템`, `Transcript` 순서를 기본으로 합니다.
- 화자명이 확정되지 않은 경우에는 `Speaker 1`, `Speaker 2` 형태를 허용합니다.
- 전사 원문은 가능한 한 원문 흐름을 유지합니다.
- 요약 영역은 전사 내용을 압축한 결과만 적고, 전사 원문을 복붙하지 않습니다.

## 출력 템플릿

```markdown
---
title: 회의 제목
date: 2026-04-11
source_file: /path/to/audio.m4a
output_file: /path/to/output/2026-04-11-회의-제목.md
language: ko
status: draft
speaker_count: 3
duration: 48m12s
transcript_method: faster-whisper
speaker_method: pyannote.audio
summary_method: ollama
tags:
  - 회의록
  - 팀미팅
---

# 회의 제목

## 메타데이터

- 날짜: 2026-04-11
- 시간: 14:00-14:48
- 참석자: 홍길동, 김민수, 이지은
- 원본 파일: `2026-04-11-회의-제목.m4a`
- 처리 상태: 완료

## 요약

- 이번 회의에서는 MVP 범위와 출력 포맷을 확정했다.
- 전사는 로컬 STT를 우선 적용하고, 화자 분리는 선택 기능으로 둔다.
- 결과물은 Markdown 파일 하나로 저장한다.

## 결정사항

- v0는 로컬 처리 중심으로 구현한다.
- 실시간 전사와 실시간 화자 분리는 MVP에서 제외한다.
- 출력 결과는 파일명 규칙에 따라 저장한다.

## 액션 아이템

- 홍길동: 샘플 오디오 3개를 준비한다.
- 김민수: STT 후보를 비교한다.
- 이지은: Markdown 템플릿 피드백을 정리한다.

## Transcript

### Speaker 1

오늘은 출력 포맷부터 먼저 고정하겠습니다.

### Speaker 2

좋습니다. 전사 결과와 요약 결과를 한 파일에 넣는 방식이 읽기 쉽겠습니다.

### Speaker 1

화자명이 확정되면 Speaker 이름만 바꾸고 본문 구조는 유지하면 되겠습니다.
```

## 메타데이터 필드 정의

메타데이터는 YAML front matter 기준으로 작성합니다. 자동 처리와 수동 검토를 모두 고려해 필드는 가능한 한 단순하게 유지합니다.

### 필수 필드

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `title` | string | 회의 제목입니다. 파일 상단 제목과 같아도 됩니다. |
| `date` | string | 회의 날짜입니다. `YYYY-MM-DD` 형식을 사용합니다. |
| `source_file` | string | 입력된 원본 오디오 파일 경로입니다. |
| `output_file` | string | 생성된 Markdown 파일 경로입니다. |
| `language` | string | 전사 언어입니다. 기본값은 `ko`입니다. |
| `status` | string | 처리 상태입니다. 예: `draft`, `reviewed`, `final`. |
| `speaker_count` | number | 감지된 화자 수입니다. 확정 불가 시 추정값을 넣습니다. |
| `duration` | string | 원본 오디오 길이입니다. `48m12s` 같은 읽기 쉬운 문자열을 사용합니다. |
| `transcript_method` | string | 전사에 사용한 엔진명입니다. |
| `speaker_method` | string | 화자 분리에 사용한 엔진명입니다. |
| `summary_method` | string | 요약 생성 방식입니다. |

### 선택 필드

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `meeting_type` | string | 정기회의, 기획회의, 회고 등 회의 종류입니다. |
| `project` | string | 관련 프로젝트명입니다. |
| `host` | string | 회의 진행자 또는 녹음 업로더입니다. |
| `attendees` | array | 참석자 목록입니다. |
| `tags` | array | 검색용 태그입니다. |
| `source_language` | string | 원발화 언어가 전사 언어와 다를 때 사용합니다. |
| `confidence` | number | 전체 결과 신뢰도 요약값입니다. |

### 상태값 권장안

- `draft`: 초안 상태입니다.
- `reviewed`: 사람이 한 번 확인한 상태입니다.
- `final`: 최종본입니다.

## Transcript 구조

Transcript는 화자 단위 블록을 기본 단위로 사용합니다. 각 블록은 아래 순서를 따릅니다.

1. 화자 제목
2. 발화 본문
3. 필요 시 보조 주석

### 권장 형식

```markdown
## Transcript

### Speaker 1

오늘은 출력 포맷부터 먼저 고정하겠습니다.

### Speaker 2

좋습니다. 전사 결과와 요약 결과를 한 파일에 넣는 방식이 읽기 쉽겠습니다.
```

### 화자명 처리 규칙

- 화자가 확정되지 않으면 `Speaker 1`, `Speaker 2`, `Speaker 3`를 사용합니다.
- 사용자가 이름을 보정한 경우, 제목을 실제 이름으로 바꾸고 본문은 그대로 유지합니다.
- 한 사람이 여러 구간에 걸쳐 발화하면 같은 화자 제목을 반복해서 사용합니다.
- 발화 중간에 길게 끊기더라도 같은 화자의 연속 발화로 묶는 것을 우선합니다.

### 긴 대화 처리

- 하나의 발화가 너무 길면 문단을 나눠도 됩니다.
- 단, 화자 제목은 첫 줄에만 둡니다.
- 요약용 메모가 섞여야 한다면 `> 메모:` 형태의 인용 블록을 사용합니다.

## 파일명 규칙

생성 파일명은 사람이 읽기 쉽고 정렬이 쉬워야 합니다.

### 권장 패턴

```text
YYYY-MM-DD-회의제목.md
```

예:

```text
2026-04-11-주간회의.md
2026-04-11-회의요약-디자인검토.md
```

### 규칙

- 날짜는 `YYYY-MM-DD` 고정 형식을 사용합니다.
- 공백은 하이픈(`-`)으로 바꿉니다.
- 특수문자는 가능하면 제거합니다.
- 같은 날짜에 여러 파일이 생기면 제목 앞부분을 더 구체적으로 적습니다.
- 중복 방지를 위해 필요하면 뒤에 순번을 붙입니다. 예: `2026-04-11-주간회의-2.md`

## 저장 위치 규칙

저장 위치는 입력 파일과 결과 파일의 관계가 바로 보이도록 정합니다.

### 기본 규칙

- 원본 오디오는 `inputs/` 아래에 둘 수 있습니다.
- 처리 결과는 `outputs/` 아래에 둡니다.
- 회의별 결과는 날짜 또는 회의명 기준으로 하위 폴더를 둘 수 있습니다.

### 권장 구조

```text
inputs/
  2026-04-11-주간회의.m4a

outputs/
  2026-04-11-주간회의/
    2026-04-11-주간회의.md
```

### 세부 규칙

- 원본 파일은 가능하면 수정하지 않습니다.
- 결과 파일은 원본 파일명과 대응되게 둡니다.
- 동일 회의에서 재처리한 경우에는 같은 폴더에 새 버전을 덮어쓰거나, 버전 번호를 붙여 보관합니다.
- 장기 보관이 필요하면 `outputs/archive/` 같은 별도 영역을 둘 수 있습니다.

### 버전 관리 권장안

- 초안은 같은 파일명으로 덮어쓸 수 있습니다.
- 외부 공유 전에는 `final` 상태로 고정합니다.
- 최종본과 초안이 함께 남아야 하면 `draft`와 `final`을 서로 다른 파일명으로 구분합니다.

## 예외 처리

- 전사만 성공하고 요약이 실패한 경우, Transcript와 메타데이터는 저장하고 요약 섹션에 실패 사유를 적습니다.
- 화자 분리가 실패한 경우, 화자 태그 없이 단일 전사 텍스트로 저장할 수 있습니다.
- 일부 구간 전사만 실패하면 실패 구간을 별도 주석으로 표시합니다.

### 실패 예시

```markdown
## 요약

- 요약 생성 실패: 입력 길이가 길어 모델 요청이 중단되었다.

## Transcript

> 전사 실패 구간: 00:31:12-00:33:05
```

## 구현 메모

- 이 문서는 v0부터 사용할 수 있는 최소 규격을 기준으로 합니다.
- 후속 단계에서 검색용 인덱스, SQLite 저장, 태그 자동 생성이 붙더라도 이 구조를 유지하는 것을 우선합니다.
- 출력 포맷 변경이 필요하면 먼저 이 문서를 갱신한 뒤 구현을 맞추는 방식을 권장합니다.
